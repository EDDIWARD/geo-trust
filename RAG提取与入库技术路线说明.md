# RAG提取与入库技术路线说明

## 1. 这一步到底在做什么

RAG 不是把所有资料一股脑塞给大模型，而是先把资料整理成“可检索、可引用、可解释”的知识底库。

对我们这个项目来说，RAG 的作用不是替代前面的指标计算，而是做两件事：

1. 给经营分析提供更可靠的外部依据。
2. 让 AI 输出建议时，能够引用“研究结论”和“案例证据”，而不是空口判断。

因此，RAG 这一步的重点不是“大模型会不会说”，而是“资料是否被提取得足够干净、结构是否足够稳定、检索是否能够命中真正相关的证据”。

## 2. 为什么不能直接把整篇论文喂给大模型

直接把整篇材料塞给模型，会有几个明显问题：

1. 成本高，材料一多就很慢。
2. 命中率低，真正有用的段落会被埋没。
3. 模型容易抓住表面词，不一定抓住研究变量和结论。
4. 新闻稿、政策稿、论文、报告混在一起时，模型会把“案例描述”误当成“普遍规律”。

所以我们要先做“资料提取”和“结构化入库”，再做后面的检索增强。

## 3. 我们采用的技术路线

我们采用“双层RAG”：

### 第一层：文档卡片

每篇资料先提炼成一张文档卡片，解决“这份资料是什么、能回答什么问题”。

文档卡片至少包含：

- 标题
- 来源文件
- 来源类型
- 主题目录
- 标签
- 适用品类
- 适用渠道
- 核心变量
- 可支持的问题
- 摘要

这一层主要用于粗筛，先把不相关的资料排出去。

### 第二层：证据切片

再把每篇资料切成若干段“证据块”，解决“具体哪一段支持这个结论”。

每个证据块至少包含：

- 所属文档ID
- 段落序号
- 文本内容
- 来源标题
- 标签
- 主题目录
- 是否命中研究结论类语句

这一层主要用于精检，给后续大模型提供可直接引用的上下文。

## 4. 为什么是“双层”而不是只做向量切片

如果只做向量切片，会有两个问题：

1. 很多材料即使切出来相似，语义也不一定是我们要的经营问题。
2. 用户问的是“该不该做礼盒”“适不适合走高端”“是不是该换渠道”，这类问题天然需要主题和变量约束。

所以我们不是单纯做“相似文本检索”，而是先做主题约束，再做证据检索。

简单说：

- 文档卡片负责“方向对不对”。
- 证据切片负责“证据够不够”。

## 5. 当前阶段怎么做

当前阶段先不引入重型向量数据库，也不依赖云端 embedding 服务，先做一个本地可跑、可验证的版本。

### 第一步：原始文件提取

支持两类输入：

- PDF
- HTML

提取目标：

- 读取正文文本
- 去掉脚本、样式、导航、噪声字符
- 保留标题、摘要、关键词、正文段落

### 第二步：文本清洗

做以下处理：

- 统一换行和空白字符
- 去掉脚本、样式、重复导航文本
- 合并断裂段落
- 保留“摘要/关键词/结论/建议/研究结果”这类结构线索

### 第三步：结构化卡片抽取

从全文中抽文档级信息：

- 标题
- 主题目录
- 来源文件
- 摘要
- 关键词
- 标签
- 适用品类
- 适用渠道

### 第四步：证据分块

将全文切为适中的块：

- 优先按标题、摘要、关键词、段落边界切
- 控制单块长度，避免过大
- 对结论类段落额外打标

### 第五步：输出标准化语料

输出三类文件：

- `documents.json`
- `chunks.jsonl`
- `insights.jsonl`

分别用于：

- 文档级粗筛
- 切片级检索
- 结论级优先召回

## 6. 为什么要单独做 insights

论文和报告里真正最值钱的，往往不是整篇正文，而是这些句子：

- “结果表明……”
- “研究发现……”
- “显著正向影响……”
- “建议……”
- “影响因素包括……”

这些句子更接近我们后面要喂给模型的“硬证据”。

所以我们会从 chunks 中再抽一层 insights，优先保留：

- 结论句
- 变量关系句
- 策略建议句

这样后面做经营分析时，可以优先引用最密的知识点。

## 7. 后续会怎么接到分析模块里

后续完整流程会是：

1. 先用现有分析模块算出商品的结构化诊断结果。
2. 把诊断结果转成检索查询，例如：
   - 高礼盒但验证偏低
   - 地理标志强但消费者感知弱
   - 适合社区团购还是礼赠渠道
3. 去 RAG 底库中先检索文档卡片，再检索证据块和 insights。
4. 把命中的证据和当前商品诊断一起喂给大模型。
5. 模型只负责“写建议”，不负责重新发明结论。

## 8. 当前版本的边界

当前版本先解决“提取”和“结构化入库”，不做以下重型工作：

- 不先接向量数据库
- 不先接远程 embedding API
- 不先做复杂 rerank
- 不先做自动摘要生成

原因很简单：先把底料清干净，比急着上复杂架构更重要。

## 9. 当前阶段的目标产物

本阶段完成后，应至少得到：

1. 一套可重复执行的提取脚本。
2. 一份稳定的结构化 RAG 语料输出。
3. 一份文档清单，知道每份资料能支持什么问题。
4. 后续可直接接入 API 检索与大模型生成的基础数据。
