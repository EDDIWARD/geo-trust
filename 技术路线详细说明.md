# 技术路线详细说明

## 一、什么是“技术路线”

对于这个项目来说，“技术路线”不是一堆技术名词，而是：

**我们准备用什么方式，把这个作品一步一步做出来。**

它主要回答四个问题：

1. 我们到底要做什么
2. 这个系统分成哪几部分
3. 每一部分分别用什么技术实现
4. 先做什么，后做什么，最后怎么连成完整作品

放到本项目里，可以把技术路线概括成一句话：

**以农户端采集真实产地信息为起点，以后端进行校验和记录为核心，以消费者扫码验真为入口，以管理端大屏展示时空异常为亮点，构建一套基于地理围栏和安全校验的农特产品可信溯源系统。**

更直白一点说，就是下面这条链路：

**登记真实产地 -> 生成唯一二维码 -> 消费者扫码验真 -> 后端识别异常 -> 管理端可视化展示**

---

## 二、推荐的整体技术路线

我建议比赛版本采用这条主路线：

- 农户登记端：`Android 原生 App`
- 消费者验真端：`H5 页面`
- 管理展示端：`Web 大屏`
- 后端服务：`FastAPI`
- 数据库：`SQLite 起步，必要时换 MySQL`
- 地图展示：`ECharts + 地图底图`
- 安全机制：`随机 token + 后端验签 + 异常扫描检测`

这条路线的优点：

- 结构清晰
- 工作量可控
- 展示效果强
- 地理和安全都能讲出亮点
- 不会因为上太重的技术导致最后做不完

---

## 三、整个系统怎么工作

这个系统可以理解成 4 个部分。

### 1. 农户端

农户端的任务是：

**只有在合法产区内，才能登记商品并生成二维码。**

它主要负责：

- 获取当前位置
- 判断是否存在虚拟定位风险
- 把商品信息和定位信息提交给后端
- 后端校验通过后生成二维码
- 返回登记成功结果

它解决的是“源头信息是否可信”的问题。

### 2. 消费者端

消费者端的任务是：

**消费者扫码后，能看到商品是不是可信、是不是第一次扫码、有没有异常风险。**

它主要负责：

- 扫码打开 H5 页面
- 调用后端接口查询商品信息
- 展示商品产区、登记时间、地图位置、扫码状态
- 如果用户授权定位，就上传当前扫码地点
- 展示风险结果，比如“首次扫码”“已开封”“疑似异常扫码”

它解决的是“消费者怎么感知真假”的问题。

### 3. 管理端

管理端的任务是：

**把系统运行状态和异常情况直观展示出来。**

它主要负责：

- 展示今日登记数、扫码数、异常数
- 展示地图上的产区点、扫码点、异常红线
- 展示异常事件趋势图
- 展示实时日志
- 支持查看某个商品或某次异常详情

它解决的是“系统有没有技术感、有没有展示冲击力”的问题。

### 4. 后端

后端是整个系统的大脑。

它主要负责：

- 接收农户登记请求
- 校验坐标是否合法
- 生成商品 token 和二维码
- 保存商品数据和扫码记录
- 判断扫码状态变化
- 计算是否存在异地异常
- 给消费者端和管理端提供接口

它解决的是“逻辑统一放在哪里、怎么统一管理”的问题。

---

## 四、为什么推荐这条路线

这套技术路线不是为了堆技术，而是因为每个端的角色不同，所以选择不同的实现方式。

- 农户端要尽量可信，所以适合原生 App
- 消费者端要低门槛，所以适合 H5
- 管理端重点是展示，所以适合 Web 大屏
- 后端重点是快、稳、接口清楚，所以适合 FastAPI

这是一种“按任务选技术”的思路，而不是“技术越多越好”。

## 五、地理模块详细设计

地理这一块不是装饰，而是这个项目的核心之一。

它主要解决两个问题：

- 农户是不是真的在合法产区里
- 消费者扫码地点是否出现异常时空变化

所以地理模块建议拆成 4 层来设计。

### 1. 产区在系统里怎么表示

首先要回答的问题是：

**合法产区的数据，在系统里怎么存。**

#### 方案 A：圆形范围

做法：

- 设置一个中心点
- 再设置一个半径
- 落在半径范围内就算合法

优点：

- 最简单
- 实现快
- 很适合快速做原型

缺点：

- 不够真实
- 真正的产区边界通常不是圆形
- 答辩时会显得比较粗糙

#### 方案 B：矩形范围

做法：

- 设置经纬度最大值和最小值
- 点落在这个矩形框里就算合法

优点：

- 实现简单
- 比圆形更容易对边界做控制

缺点：

- 仍然不够真实
- 表达能力一般
- 会把一些不属于产区的区域也包含进去

#### 方案 C：多边形 Polygon

做法：

- 提前准备产区边界点
- 把它保存成多边形坐标集合
- 农户上传坐标后判断这个点是否落在多边形内部

优点：

- 最符合真实地理边界
- 展示时更专业
- 很适合结合测绘背景来讲
- 可以突出“物理定界”的概念

缺点：

- 比圆形和矩形稍复杂
- 需要提前整理边界点数据

#### 建议选择

比赛版本建议采用：`多边形 Polygon`

原因：

- 难度可控
- 专业感明显更强
- 和“测绘 + 信安”的项目定位更匹配

---

### 2. 农户定位怎么实现

农户端定位直接决定源头数据是否可信。

#### 方案 A：Web / 小程序定位

优点：

- 开发快
- 跨平台
- 门槛低

缺点：

- 更容易被模拟
- 权限和精度不稳定
- 很难做像样的反作弊
- 不适合拿来作为可信登记入口

#### 方案 B：Android 原生定位

优点：

- 能拿到更完整的定位信息
- 更适合做 mock 定位检测
- 更容易做基础风控
- 更符合“源头登记必须可信”的要求

缺点：

- 开发量比 Web 大
- 需要有 Android 开发能力

#### 建议选择

比赛版本建议采用：`Android 原生定位`

原因：

你们最关键的一步是“生成二维码之前，先保证登记地点尽可能真实”，这一点 Web 端很难支撑。

---

### 3. 产区校验怎么实现

当农户端把坐标传给后端之后，后端要判断这个点是不是在合法产区范围内。

#### 方案 A：自己实现射线法

优点：

- 原理清楚
- 可以在答辩时讲算法过程
- 更有“自己实现”的感觉

缺点：

- 容易出现边界 bug
- 细节处理麻烦
- 调试时间更长

#### 方案 B：使用现成地理库

例如：

- `shapely`

优点：

- 稳定
- 节省时间
- 更适合把精力放在系统联动上
- 很适合比赛项目

缺点：

- “自己写算法”的成分少一点

#### 建议选择

比赛版本建议采用：`shapely` 这类成熟地理库

答辩时可以这样解释：

**底层逻辑本质上仍然是“点是否落在多边形内”，工程上采用成熟库来提高稳定性。**

---

### 4. 地图展示怎么做

地图展示分为两类需求：

- 消费者端的小地图
- 管理端的大屏地图

#### 方案 A：直接使用地图 SDK 做全部展示

例如：

- 高德地图
- 百度地图

优点：

- 地图能力强
- 点位、围栏、路径等功能完整
- 很接近真实业务系统

缺点：

- 接入会更重
- 前端复杂度较高
- 不一定最适合做比赛大屏风格

#### 方案 B：ECharts 为主，地图服务辅助

优点：

- 很适合比赛可视化
- 地图点位、异常红线、趋势图风格统一
- 更容易做出“态势感知大屏”的感觉

缺点：

- GIS 交互能力不如专业地图 SDK
- 不适合特别复杂的地图分析

#### 建议选择

管理端建议采用：`ECharts 为主，地图服务辅助`

原因：

你们的重点不是做一个完整 GIS 平台，而是做一个“异常感知与防伪展示系统”。

## 六、安全模块详细设计

安全这一块是你们另一个核心亮点，但这里要坚持一个原则：

**比赛作品的安全设计，不追求理论上最强，而追求能实现、能演示、能讲清楚。**

你们主要是在防三类问题：

- 防伪造登记
- 防篡改溯源信息
- 防复制二维码后批量造假

安全模块可以拆成 4 层。

### 1. 二维码里放什么

这是最关键的一层。

#### 方案 A：二维码里直接放完整信息

例如：

- 商品名
- 产地
- 时间
- 经纬度

优点：

- 看起来信息完整
- 扫码后不太依赖后端

缺点：

- 关键信息容易暴露
- 容易被复制和分析
- 状态逻辑难以动态调整
- 安全性实际并不好

#### 方案 B：二维码里只放随机 token

优点：

- 实现最简单
- 更稳
- 真正的数据都保存在后端
- 状态逻辑可以全部放在后端动态控制
- 二维码即使被拍照复制，也只是复制了一个入口

缺点：

- 必须依赖后端查询
- 没有离线能力

#### 建议选择

比赛版本强烈建议采用：`二维码只放 token`

这也是最像真实系统的做法。

---

### 2. 要不要做签名

在随机 token 之外，还可以进一步做签名校验。

#### 方案 A：只有随机 token，不做签名

优点：

- 最简单
- 开发快
- 比赛里已经能用

缺点：

- 安全亮点相对弱一点
- “防篡改”这一块说服力没那么强

#### 方案 B：随机 token + 服务端 HMAC 签名

例如对下面这些字段做签名：

- 商品 ID
- 产区 ID
- 生成时间
- 坐标摘要

优点：

- 能突出“关键字段可验签”
- 答辩时更容易讲清楚“防篡改”
- 实现并不算特别复杂

缺点：

- 逻辑会多一点
- 需要维护密钥

#### 建议选择

推荐采用：`随机 token + 服务端 HMAC 签名`

推荐表述：

**二维码只承载商品唯一 token，关键溯源字段由服务端签名，后续验证时若发现签名不一致，则判定数据可能被篡改。**

---

### 3. 农户端防虚拟定位怎么做

这部分是“源头可信”的关键。

#### 方案 A：基础检测

例如：

- 判断位置是否带 mock 标记
- 判断是否为模拟器
- 判断是否处于调试环境
- 判断轨迹跳变是否异常

优点：

- 开发量适中
- 足够支撑比赛演示
- 能体现“不是纯网页填表”

缺点：

- 防御不绝对
- 仍可能被专业手段绕过

#### 方案 B：上更重的加固方案

例如：

- NDK 封装签名逻辑
- 设备完整性校验
- 更底层的环境检测

优点：

- 安全故事更强
- 更符合信安方向

缺点：

- 开发工作量大
- 联调复杂
- 风险高
- 很容易拖慢整体进度

#### 建议选择

比赛第一版建议采用：`基础检测`

原因：

先把“能拦一部分虚拟定位”的效果做出来，比追求很重的加固更重要。  
如果后面还有时间，再继续补 NDK 和更强的环境检测。

---

### 4. 克隆码和异地异常怎么判断

这是最适合做演示的安全功能之一。

核心思路：

- 记录每次扫码的时间和地点
- 新扫码到来时，与上一次扫码进行比较
- 计算时间差、距离差、推定移动速度
- 如果速度明显不合理，就认定为异常

#### 方案 A：简单规则判定

例如：

- 短时间跨超远距离 -> 异常
- 多次高频扫码 -> 可疑
- 同一码多地短时出现 -> 高风险

优点：

- 很直观
- 好实现
- 好解释
- 非常适合比赛

缺点：

- 精度有限
- 不如复杂模型那么灵活

#### 方案 B：风险评分模型

例如综合：

- 时间
- 距离
- 扫码频次
- 设备信息
- 历史行为

计算总风险分

优点：

- 更像真实风控系统
- 后续扩展空间大

缺点：

- 复杂
- 规则难调
- 比赛里未必有必要

#### 建议选择

比赛版本建议采用：`简单规则判定`

推荐答辩表达：

**同一个码在极短时间内出现在相距数百甚至数千公里的地点，系统判断其不可能由正常流通行为产生，因此触发克隆码告警。**

## 七、关键技术方案比较汇总

为了体现方案思考深度，可以把关键技术的对比单独整理出来。

### 1. 地理围栏实现方案比较

方案一：圆形/矩形围栏

- 优点：实现简单，适合快速原型
- 缺点：对真实产区边界表达不准确，专业性不足

方案二：多边形围栏

- 优点：更符合真实地理边界，能体现测绘专业特色，适合答辩展示
- 缺点：需要提前准备边界点数据，实现略复杂

最终选择：`多边形围栏`

原因：

在实现难度可控的前提下，能明显提升项目真实性和专业性。

### 2. 农户端定位方案比较

方案一：Web / 小程序定位

- 优点：开发快，跨平台
- 缺点：容易受模拟定位和浏览器环境影响，不适合作为可信登记入口

方案二：Android 原生定位

- 优点：更适合获取稳定定位信息并进行基础反作弊检测
- 缺点：开发工作量较大

最终选择：`Android 原生定位`

原因：

源头登记的可信性比开发便捷性更重要。

### 3. 二维码内容方案比较

方案一：二维码直接存储完整溯源信息

- 优点：数据直观，可脱离后端查看部分信息
- 缺点：易暴露关键信息，难以动态控制状态

方案二：二维码仅存储随机 token

- 优点：安全性更高，状态逻辑可后端动态调整，实现更稳
- 缺点：依赖后端查询

最终选择：`随机 token`

原因：

更符合比赛项目在安全性和工程可实现性之间的平衡。

### 4. 安全校验方案比较

方案一：仅使用随机 token

- 优点：简单、稳定、易实现
- 缺点：安全亮点略弱

方案二：随机 token + 服务端 HMAC 签名

- 优点：能够提供更明确的防篡改能力，答辩时更有说服力
- 缺点：实现略复杂

最终选择：`随机 token + 服务端 HMAC 签名`

原因：

实现成本不高，但能明显增强项目亮点。

## 八、比赛版推荐实施顺序

为了避免项目失控，建议按三层推进。

### 第一阶段：先做基础闭环

先完成：

- 农户端提交商品登记
- 后端判断是否在合法产区
- 后端生成 token 和二维码
- 消费者扫码查看商品信息
- 后端记录扫码次数并返回状态

这一阶段完成后，项目就已经“跑起来了”。

### 第二阶段：补展示效果

继续完成：

- 管理端地图大屏
- 展示产区点、扫码点、异常红线
- 展示趋势图和日志
- 增加商品详情和异常详情

这一阶段完成后，项目就已经“好看了”。

### 第三阶段：补安全亮点

最后补充：

- 农户端基础虚拟定位检测
- 后端关键字段签名校验
- 异地扫码异常识别
- 重复扫码风险提示

这一阶段完成后，项目就已经“有技术说服力了”。

## 九、推荐的比赛版技术路线结论

如果要确定一条最稳、最适合比赛的路线，建议最终采用下面这个版本：

- 农户端使用 `Android 原生 App`，负责可信登记与基础定位风控
- 消费者端使用 `H5 页面`，负责扫码验真与状态展示
- 管理端使用 `Web 大屏`，负责地图、趋势图、日志和异常展示
- 后端使用 `FastAPI`，负责业务逻辑、异常判断和接口联动
- 数据库先使用 `SQLite`，保证部署简单、演示方便
- 地理模块采用 `多边形围栏 + 点在多边形内判断`
- 安全模块采用 `随机 token + 服务端 HMAC 签名 + 异地扫码检测`

这条路线的优势是：

- 能做完
- 能展示
- 能讲清楚
- 能体现专业特色

## 十、可直接用于方案书的总结表述

本项目技术路线以“地理可信 + 状态可信 + 异常可视”为主线。系统采用 Android 原生农户端、H5 消费者端、Web 管理端与 FastAPI 后端相结合的总体架构。农户端负责采集商品登记信息与定位信息，后端基于预设产区多边形进行地理围栏校验，通过后生成商品唯一 token 与二维码；消费者扫码后进入 H5 页面，由后端返回商品溯源信息、当前扫码状态及风险提示；管理端通过地图、趋势图与实时日志展示商品流转情况及异常事件。安全机制上，系统采用“随机 token + 服务端签名 + 异常扫码检测”的轻量方案，在保证比赛阶段可实现性的同时，突出可信产地校验与克隆码识别这两个核心亮点。
