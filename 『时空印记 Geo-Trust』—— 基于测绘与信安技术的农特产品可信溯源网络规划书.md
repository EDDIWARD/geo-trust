# 『时空印记 Geo-Trust』—— 基于测绘与信安技术的农特产品可信溯源网络规划书

## 〇、 项目概况

大家平时看到的“特产地图”或者“溯源电商”，本质上只是一个**信息展示网页**。它们最大的痛点是：**无法保证源头数据的真实性**（农民可以乱填产地，黑产可以随意复制溯源二维码）。

我们这次参加计算机设计大赛，**不打算只做一个普通的电商网站，我们要利用团队“测绘 + 信安”的专业背景，做一套打击农业造假黑灰产的“数字安全基础设施”。**

我们的核心理念是：**零信任架构下的时空强制绑定**。

- **测绘组负责“物理定界”：** 划定高精度真实产区，不在产区内打卡，系统绝对不认。
- **信安组负责“数据上锁”：** 用底层加密和抗逆向技术，把测绘得到的位置数据焊死，让黑灰产无法篡改，并在后端抓捕物理克隆的假货。

## 一、 具体化场景流程与演示剧本（比赛答辩时我们演什么？）

为了在答辩的 5 分钟内抓住评委眼球，我们要在现场进行一场**“攻防演练”**。商品流转的每个环节，我们都会配套一个对抗演示：

### 1. 源头登记环节：农田打卡发货

- **正常演示：** 农户端点击“打卡生码”。系统调用高德地图，精准定位到当前位置。系统提示“定位坐标位于五常大米核心产区，时空防伪签名成功”，生成防伪二维码。
- **攻击模拟：** 我们现场打开一个“虚拟定位/篡改GPS”软件，把位置改到外地，再次点击打卡。系统立刻拦截：“检测到环境异常或坐标偏离核心产区，拒绝生成溯源码”（展示端侧反欺诈能力）。

### 2. 物流运输环节：数据黑盒

- **讲述重点：** 在此环节，商品包装上的二维码内包含了“加密的经纬度、时间戳、硬件指纹”。任何人（包括快递员）都无法在不破坏数字签名的情况下修改产地数据。

### 3. 消费者验真环节

- **正常演示：** 评委用自己的手机扫码，打开一个 H5 网页。网页像杀毒软件一样跳出三个绿勾：`[√]密码签名无异常`、`[√]地理围栏校验通过`、`[√]未发现物理克隆风险`。随后展示一张地图，标明该特产真实的出生点和时间。

### 4. 终极防御环节：拦截批量造假

- **攻击模拟：** 我们假设这个真码被黑灰产复印了 1000 份。台上的同学 A 扫一次码，同时台下的同学 B 利用代理或者虚拟定位，假装在 2000 公里外的城市也扫了这个同样的码。

- **演示效果：** 我们的“管理端大屏”地图上出现一条红线连接两地。系统日志滚动：“触发时空悖论，判定为恶意实体克隆”

  同时，二次扫描的消费者端显示“该产品已被开封，如若不是本人操作请鉴别真假 ”

## 二、 现有溯源系统的致命不足（为什么我们能赢？）

市面上的竞品大多存在“Garbage In, Garbage Out（垃圾进，垃圾出）”的问题。

1. **源头数据靠“人工填”：** 现有的溯源系统（如很多地方政府的农产品追溯平台），其产地信息往往是商贩在后台网页手动录入的。这完全防君子不防小人，导致“洗澡蟹”、“外地米”泛滥。

2. **溯源码极易被物理克隆：** 传统的溯源码只是一个静态链接。造假者买一盒真货，把码拍下来批量印刷，消费者扫出来的依然是真的信息，造成了极大的消费者信任危机。

   > *参考依据：《商务部关于重要产品信息化追溯体系建设的指导意见》中明确指出，需强化追溯信息的安全保护，严防信息篡改和伪造。目前的市场现状是，前端的业务逻辑做得很漂亮，但底层的安全和防伪验证几乎为零。*

## 三、 系统端侧设计方案（三个端）

### 1. 农户登记端（建议：Android 原生应用）

- **为什么不用小程序/Web？** 因为 Web 端的 Geolocation 极易被浏览器插件或模拟器欺骗，且 JS 代码容易被反编译。我们需要一个能做**底层加固**的环境。
- **功能：** 获取高精度 GPS 坐标、获取网络时间、融合硬件指纹，生成并打印加密二维码。

### 2. 消费者验真端（建议：Mobile Web / H5）

- **为什么用 Web？** 消费者极其讨厌为了扫个码去下载 APP。H5 网页随扫随开，体验最好。
- **功能：** 请求浏览器的地理位置权限（话术包装成“为了给您提供时空防伪安全检测”），解密二维码并展示溯源地图。

### 3. 管理员态势感知大屏（建议：Web PC 端）

- **功能：** 用 ECharts 渲染中国地图。不展示无聊的销售数据，专门展示“拦截了多少次伪造打卡”、“抓获了多少个异地克隆扫码”，用满屏的防御日志凸显技术硬核感。

## 四、 核心技术落地指南（具体怎么敲代码？）

### 技术点 1：地图调用与地理围栏划定（测绘组主导）

- **怎么做：** 在后台调用高德地图或百度地图的 JavaScript API。测绘同学可以通过工具在地图上绘制代表“核心产区”的多边形（Polygon），系统将其转化为一组经纬度坐标点集（如 GeoJSON 格式）存入数据库。
- **打卡判定逻辑：** 当农户 APP 上传了一个坐标点 $(x, y)$，后端使用经典的**“射线法（Ray Casting Algorithm）”**判断该点是否在预设的多边形内。

### 技术点 2：抗逆向的数据加密防篡改（信安组主导）

- **怎么做：** 绝不在易被反编译的 Java 层拼接打卡数据。信安同学可以使用 **C 语言**编写核心的加密算法（例如 HMAC-SHA256），将 `经纬度 + 时间戳 + 动态随机盐` 进行哈希签名，然后编译成 Android 的 `.so` 动态链接库。
- **防御效果：** 灰产团伙如果企图编写脚本伪造打卡数据，就必须动用 x64dbg 或 IDA 等工具去逆向分析底层的汇编指令。这直接将造假成本拉升到了极其专业的软件对抗级别。

### 技术点 3：服务端“时空异常检测”算法（抓克隆码）

- **怎么做：** 数据库中记录每个二维码上一次被扫描的 `[经纬度, 时间]`。当新扫码请求到达时，后端利用 **Haversine 公式**计算两次扫码地点的球面距离，并除以时间差得出“移动速度”。
- **熔断逻辑：** 如果速度超过了物理常理（比如大于商用客机速度 1000km/h），则 100% 判定物理码被批量克隆，立即触发大屏报警并封禁。

建议做成 **“1 个 Python 后端 + 2 个前端页面 + 1 个 Android 农户端”**。

- 在产区内 → 允许生成码

## 问题 6：如何加密？

比赛阶段不要上太重的方案，建议分成两层理解：

### 方案 A：最简单可做版

二维码里只放随机 token，不直接暴露真实产地信息。
真正的数据都存在后端数据库里。

优点：

- 实现最简单
- 安全上也够比赛演示
- 二维码被看到，也不能直接伪造数据

### 方案 B：再加一层签名

后端生成 token 时，再做一次签名，比如：

```text
sign = HMAC(secret, 商品ID + 时间戳 + 坐标)
```

后端验证时重新算一遍。
这样别人就算知道规则，也没法随便伪造合法码。

比赛阶段我更建议你们走 **“随机 token + 后端存真数据 + 可选 HMAC 签名”** 这个路线，简单而且稳。Redis 官方文档也说明了键可以设置过期时间，适合做短期 token。([Redis](https://redis.io/docs/latest/commands/expire/?utm_source=chatgpt.com))

## 问题 8：如何让二维码第一次扫描显示“已开封”，第二次之后显示别的？

数据库里存：

- `scan_count`
- `first_scan_time`
- `last_scan_time`

- `scan_count = 0`：显示“首次扫码，商品未开封”

如果你们想更细一点，可以再加：

- 第二次扫码且地点接近第一次 → 正常重复查看

这个就已经很有演示效果了。

## 问题 12：如何判断“克隆码”或“异地异常”？

最简单就是比较两次扫码的：

- 时间差
- 地点差

后端拿到两个坐标后算距离，再除以时间差，得到一个“推定移动速度”。

如果速度离谱，比如几分钟跨几百上千公里，就可以判定为异常。

比赛阶段不需要搞复杂模型，简单规则就够：

- 短时间超远距离 → 异常
- 多次高频扫码 → 可疑
- 多地同时扫码 → 高风险

## 问题13：如何让二维码第一次扫描显示“已开封”，第二次及之后显示“异常”？

### 简单做法

不要把“是否开封”写死在二维码里，而是让二维码只存一个**商品唯一ID**或**短链接**。

扫码后流程是：

1. 用户扫二维码
2. 打开你们的后端接口
3. 后端查询数据库里这个商品的扫描记录
4. 根据扫描次数返回不同状态

### 状态设计可以很简单

- **0次扫描**：显示“首次扫码，商品状态正常”
- **1次扫描**：显示“该产品已被开封”
- **2次及以上**：显示“该商品已存在重复扫码记录，请注意鉴别真假”

### 数据库里要存什么

一张 `product` 表，一张 `scan_record` 表就够了。

#### product

- id
- 商品编号
- 二维码token
- 创建时间
- 当前状态

#### scan_record

- id
- product_id
- 扫码时间
- 扫码地点
- 扫码设备信息
- 扫码结果（正常/重复/异常）

### 为什么这样做

因为二维码本身是静态的，**变化逻辑必须放在后端**。
这样你们以后想改成“第一次正常，第二次提示开封，第三次报警”也很方便。

## 问题14：如何生成对应二维码？

### 简单做法

后端生成一个唯一 token，然后把它编码成二维码图片。

例如：

- 商品ID：1001
- token：`a8f4c9e2...`
- 二维码内容：`https://你的域名/verify?a=a8f4c9e2`

### Python 方案

可以直接用：

- `qrcode`
- `Pillow`

生成二维码图片，然后保存到服务器本地或对象存储里。

### 为什么不要直接把所有信息塞进二维码

因为如果二维码里直接明文放：

- 产地
- 时间
- 经纬度

那就太容易被复制和分析了。

比赛里更稳的方案是：

> 二维码里只放一个 token，真正的信息去后端查。

这样实现简单，也更像真实系统。

## 问题15：如何防止别人随便伪造一个二维码？

### 简单做法

二维码里的 token 不要是简单自增数字，要做成**随机字符串**，长度尽量长一点。

比如：

- UUID
- 32位随机字符串
- 带签名的 token

### 推荐方案

后端生成：

- `product_id`
- `random_token`
- `sign`

最后二维码内容只存 `random_token` 或一个完整短链接。

数据库里保存 token 和商品的映射关系。
扫码时，如果 token 查不到，就直接提示：

> 无效二维码，疑似伪造

### 比赛里够不够

够了。
你们没必要上特别复杂的密码协议，**随机 token + 后端校验**已经能做出效果。

## 问题16：如何给消费者提供定位服务？

### 简单做法

消费者扫码进入 H5 页面，然后浏览器请求定位权限。

前端调用浏览器定位接口，获取：

- 经度
- 纬度
- 精度
- 时间

然后传给后端。

### 作用

消费者定位主要不是为了“证明消费者是谁”，而是为了：

- 记录扫码位置
- 做异地扫码检测
- 在地图上显示扫码轨迹

### 注意

消费者端定位不需要做太强的反作弊，因为你们比赛重点是：

- 展示时空异常
- 做可视化
- 体现“这个码在哪被扫过”

所以消费者端定位只要能拿到位置并上传就够了。

## 问题17：如何给农户端提供定位服务？

### 简单做法

农户端建议用 **Android 原生 App**。

因为这里只有原生端更适合：

- 取 GPS
- 做常见虚拟定位检测
- 限制只有在产区内才能登记

### 农户端流程

1. 点击“登记生成码”
2. 获取当前位置
3. 检查是否命中虚拟定位风险
4. 把位置发给后端
5. 后端判断是否在合法产区
6. 合法才允许生成二维码

### 这里最关键的点

农户端定位不是为了“展示地图”，而是为了：

> **作为生成二维码的前置条件**

## 问题18：如何判断是否在合法产区内？

### 简单做法

给每个产区提前存一组多边形坐标点，然后后端做“点是否在多边形内”的判断。

### 实现方式

你们可以：

- 前端地图上画出产区范围
- 后端存成 GeoJSON 或经纬度数组
- Python 用现成库判断点是否在区域内

### Python 可选方案

- `shapely`
- `geopandas`（如果你们想更重一点）
- 或自己写简单射线法

### 比赛建议

直接用 `shapely` 最省事。
逻辑也好讲：

> 农户打卡点如果不在预设产区多边形内，就拒绝生成二维码。

## 问题19：如何实现“异地扫码判定”？

### 简单做法

记录同一个商品的每一次扫码时间和地点。
当出现新扫码时，把它和上一次扫码比较：

- 时间差
- 距离差
- 推算速度

如果速度过高，比如超过 800 或 1000 km/h，就判定异常。

### 距离怎么算

用 **Haversine** 公式即可。

### 为什么这个适合比赛

因为它：

- 简单
- 直观
- 非常适合做地图红线演示

这块几乎是你们管理端可视化最值得做的功能之一。

## 问题20：如何加密？

### 比赛里建议：不要上太重的加密设计

你们的重点不是发论文级密码系统，而是做一个完整可运行项目。

### 推荐简单方案

分成两层：

#### 第一层：二维码只存 token

这本身就是一种“隐藏真实数据”的方式。

#### 第二层：后端对关键字段做签名

例如对：

- 商品ID
- 产区ID
- 生成时间
- 经纬度摘要

生成一个 HMAC 签名。

### Python 里怎么做

直接用：

- `hashlib`
- `hmac`

就够了。

### 作用

后端以后如果想验证数据有没有被改，可以重新验签。

### 比赛里怎么讲

可以说：

> 关键溯源字段在生成时由服务端签名，后续验证时若发现签名不一致，则判定数据可能被篡改。

这样已经很够用了。

## 问题25：如何实现“首次扫码正常，后续异常”的页面切换？

### 简单做法

前端不要自己判断，前端只请求后端状态。

后端接口返回：

- `status = first_scan`
- `status = opened`
- `status = risky`
- `status = fake`

前端根据状态显示不同页面文案和颜色。

### 为什么这样好

因为状态逻辑都集中在后端，方便改。
你们比赛时也更容易调试。
